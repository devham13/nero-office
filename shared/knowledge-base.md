## 0. Приоритет: лонгриды и верх выдачи (Яндекс / Google)

- **Формат по умолчанию для офиса** — **лонгрид** (глубокий материал под кластер запросов), а не короткая заметка. Узкие страницы — только если владелец **явно** ограничил объём или тип.
- **Цель поиска:** ориентация на **максимальную конкурентоспособность** в **верхней части выдачи** (в т.ч. **первая строка / топ-1** как **желаемый ориентир**). Поисковики **не гарантируют** позицию; выигрывает сочетание **полноты по интенту**, **структуры**, **E-E-A-T**, **уникальной пользы**, **технического качества** страницы и профиля сайта.
- Лонгрид для борьбы за топ: закрывать **все основные подзапросы** из Wordstat по теме, **лучше топа** по глубине и ясности, без «воды» и дублирования чужих текстов.

## 2. Объём текста: лонгрид в первую очередь (SEO 2026)

**Важно:** у Google и Яндекса **нет** официальной «нормы знаков» за страницу; ранжируют **полезность, полноту ответа, интент, структуру, E-E-A-T**, а не длину ради длины ([PR-CY, обновление 2026](https://pr-cy.ru/news/p/7342-optimalnaya-dlina-seo-teksta-chto-aktualno-v-2026); данные по Google: средняя страница в топ-10 — порядка **1447 слов** по исследованию Backlinko, при этом длина **равномерно** распределена по позициям 1–10 — значит одна цифра слов не гарантирует место).

**Вывод для офиса:** для **информационных** тем приоритет — **лонгрид**, который **покрывает кластер** сильнее конкурентов в топе; объём — **следствие** полноты, но **стартовый коридор для лонгрида** задаём **выше**, чем для «короткой страницы» (см. таблицу ниже).

### Практические ориентиры по типу материала (русский текст, **знаки с пробелами**)

| Тип страницы | Ориентир объёма | Комментарий |
|--------------|-----------------|-------------|
| Узкая страница / один запрос | **2 500–4 500** | Лендинг под один интент, без «воды» |
| Стандартная экспертная статья | **4 500–7 000** | Часто достаточно для средней конкуренции |
| **Лонгрид** (основной формат офиса: инфо, гайд, «столб») | **8 000–14 000** | Ориентир для борьбы за **верх выдачи**; H2/H3, оглавление, списки, FAQ, GEO-блоки |
| Сильная конкуренция / много подтем (цель — топ) | **14 000–20 000+** | Только если каждый блок закрывает реальный подзапрос из Wordstat; без «воды» |

**Про ~3 500 знаков:** для лонгрида, претендующего на **верх выдачи**, это **как правило недостаточно**. Без жёсткого лимита от владельца для **лонгрида по умолчанию** бери **не ниже ~8 000 знаков**; комфортный коридор **8 000–14 000**, при «тяжёлом» топе — **14 000+** по обоснованным подтемам.

**Дополнительно (GEO / AI-выдача):** длину не наращивают пустыми абзацами — добавляют **чёткие определения, списки, таблицы, FAQ** ([PR-CY](https://pr-cy.ru/news/p/7342-optimalnaya-dlina-seo-teksta-chto-aktualno-v-2026)).

## 3. Как субагенты используют этот файл

1. **Начало задачи:** прочитать разделы 1–2 и «Рабочий контекст сессии» (если заполнен). Сквозная задача «страница целиком»: если точной темы нет, сначала **Кирилл** выбирает новость дня без дублей; дальше **Коля||Артём → Женя → Артур → Алина||Борис → Наташа → Юра → Макс||Лёня**. Если сказано «без публикации», остановиться до Юры; если «только SEO», не запускать визуальную/публикационную часть.
2. **После выдачи результата:** директор или ведущий агент может **кратко** дописать в «Рабочий контекст» тему, принятые решения по объёму, ссылку на страницу — чтобы следующий субагент не терял нить.
3. **Hero + блок в статье:** **Алина** (скилл **animator-alina**) — только **первый экран**; параллельно **Борис** (скилл **animator-boris**) — **не hero**, отдельная анимация **в теле лонгрида** (продолжение или контраст сцене Алины). Алина читает **`shared/hero-animation-reference.md`** и **`shared/hero-engine-example.js`**. У каждого — **свои** `id` у canvas/script.
4. **Вёрстка всей страницы:** **Наташа** (скилл **designer-natasha**) — **hero** первым, затем **введение после hero** (второй блок: текст **слева** + декор, см. **longread-page-design-system.md**), дальше контент и **вставка блока Бориса** по якорю из handoff; стили — **`shared/longread-page-design-reference.css`**, reveal — **`shared/longread-page-reveal.js`**, конспект — **`shared/longread-page-design-system.md`**.
5. **Публикация на хостинг:** **Юра** (скилл **publisher-yura**) — выкладка; затем **проверка** публичного URL, ответ со ссылкой вида **`https://…/slug/`**; запись в **`<PROJECT_ROOT>/nero-network-office-page/shared/published-pages.md`** и **session-handoff**. В Cloud публичный домен и доступы берутся из Cursor Secrets / env vars (`PUBLIC_SITE_URL`, `WP_SITE_URL`, `FTP_*`, `SSH_*`, `REMOTE_*`). Локально можно использовать `hosting-credentials.local`, но его нельзя коммитить.
6. Путь к файлу в установленном плагине: `nero-network-office-page/shared/knowledge-base.md` (в репозитории разработки: `Neurocom/nero-network-office-page/shared/knowledge-base.md`).
7. **Типичные сбои и как их избегать (ускорение пайплайна):** перед полной страницей прочитай **`shared/agent-pipeline-pitfalls.md`** — тема `.site-main` padding, hero, FTP vs API, Task для Лёни, QA-зазор под шапкой и т.д.
